# LeetCode 147: 对链表进行插入排序 (Insertion Sort List)

## 题目描述
给定单个链表的头节点 `head` ，使用 **插入排序** 对链表进行排序，并返回排序后链表的头节点。

**插入排序的特点：**
最差时间复杂度是 $O(N^2)$，最优时间复杂度是 $O(N)$（当数组或链表几乎有序时）。空间复杂度要求在链表节点上原地修改为 $O(1)$。

---

## 循序渐进的解法

### 方法一：转化为数组后排序（简单暴力法）
这种解法并不是链表插入排序的主旨，但作为做题时快速拿到 Accept 的“兜底”暴力解法，非常简单好写。

**思路：**
1. 遍历一次链表，把所有节点的值塞入到一个数组 / 列表中。
2. 对数组进行自身底层的排序操作（如 Python 的 `sort()`，底层通常是归并排序改编的 Timsort，时间复杂度 $O(N \log N)$）。
3. 再次遍历链表，依次用数组排序好的数值覆盖链表节点原本的值。

**复杂度分析：**
- **时间复杂度**：$O(N \log N)$。遍历链表 $O(N)$，数组排序 $O(N \log N)$，二次遍历重新赋值 $O(N)$。
- **空间复杂度**：$O(N)$。需要额外一个数组来存储所有数据。虽然不是 $O(1)$ 的解法，但通俗易懂。

---

### 方法二：基础版链表插入排序（符合题意的标准解法）
要真正符合“插入”定义并且不使用数组，需以 $O(1)$ 的额外空间做原地处理。

**思路：**
将链表打断为“已排序部分”和“未排序部分”。
1. 引入一个 **哑节点 (dummy node)** 放在原来链表的头部（好处是当我们需要在第一个节点前插入新数据时，可以免去很多边界情况的判断，这是单链表操作的老套路了）。
2. 让 `curr` 指向**待排序**的第一个节点（我们先截断第一个节点作为已排序部分）。每次拿出 `curr`，我们都从 `dummy` 开始向后寻找它正确的归属位置。
3. 找到插入的前驱节点 `prev` 后（即满足 `prev.next.val > curr.val` 或 `prev.next == None` 时），将 `curr` 插入到 `prev` 和 `prev.next` 之间，然后进入下一轮循环。

这是一种能Work的版本，缺点是：**不论链表原先是不是有序的，或者部分连续有序的，每个节点依然要每次“傻头傻脑”地从头开始寻找位置。**

**复杂度分析：**
- **时间复杂度**：始终为 $O(N^2)$。嵌套的两层遍历，就算原本就有序依然要每次扫描。
- **空间复杂度**：$O(1)$。在原表上修改指针。

---

### 方法三：优化版链表插入排序（实际应用中最优解）
面对一串类似 `1 -> 2 -> 3 -> 4 -> 5` 已经排好序的链表或局部排好序的链表，方法二依然要退化成 $O(N^2)$。怎么优化？我们维护一个“尾哨兵”来记录已排序内容的末尾。

**思路：**
1. 依然使用哑节点 `dummy`。
2. 维护两个指针：`last_sorted` 表示 **已排序部分的最后一个节点**，`curr` 表示 **当前待插入的节点**（初始化 `last_sorted = head`, `curr = head.next`）。
3. 当我们要给 `curr` 找位置时，**先比较它与已排序尾部 `last_sorted` 的大小**：
   - 如果 `last_sorted.val <= curr.val`，说明 `curr` 应该在的位置就是紧接在尾巴后面，完美！原来就排好了，我们只需要向后走一步 `last_sorted = last_sorted.next` 即可。
   - 如果 `last_sorted.val > curr.val`，此时这部分顺序坏了，说明必须插到前面去，我们才退回到 `prev = dummy` 开始向后找，把 `curr` 放进刚好合适的前驱节点中。

**复杂度分析：**
- **时间复杂度**：最坏情况下（例如纯逆序链表），还是无法避免比较，为 $O(N^2)$。但在最好情况（天然有序或接近有序时），只需走一次外循环比较，退化为 $O(N)$。平均时间大幅下降。
- **空间复杂度**：$O(1)$。利用原链表的引用连接。
