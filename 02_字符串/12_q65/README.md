# LeetCode 65: 有效数字 (Valid Number) 详细解析

这道题是经典的字符串处理问题。题目要求判断一个字符串是否可以表示一个合法的数字。虽然规则看上去有点多，但只要我们把规则拆解开，其实非常符合直觉。

我们从最简单粗暴的正则法开始，重点讲解最适合面试的**状态标记法**，最后再了解一下偏理论的**有限状态自动机(DFA)**。

---

## 1. 简单粗暴法：正则表达式 (Regex)

**核心思路：**
Python的 `re` 模块可以用一行代码解决这种规则死板的字符串匹配问题。我们只需要把题目对“有效数字”的定义翻译成正则表达式即可。

**正则拆解：**
* `^[+-]?`：可选的正负符号。
* `(\d+\.\d*|\.\d+|\d+)`：主数部分，包含三种情况：
   1. `\d+\.\d*`：数字跟小数点，例如 `1.` 或 `1.2`。
   2. `\.\d+`：小数点跟数字，例如 `.1`。
   3. `\d+`：纯数字，例如 `1`。
* `([eE][+-]?\d+)?$`：可选的指数部分，包含 `e` 或 `E`，跟着可选符号，最后必须跟着纯数字。

**优点：** 代码极短。
**缺点：** 在面试中，很多面试官不允许使用完全依赖内置库或正则的方式，因为这没有展现你的逻辑分析能力。

---

## 2. 最适合面试：状态标记法 (Flag-based Simulation) ⭐强烈推荐

对于这道题，**状态标记法（或者叫标志位法）绝对是首选**。
为什么？因为它逻辑清晰，不需要去死记硬背复杂的表格，就像是在对人讲话一样自然。你只需要设身处地地想想：当我们从左到右读一个数字时，如果看到了小数点、字母或加减号，它们满足什么条件才是合法的？

### 💡 小白也能懂的逻辑拆解

我们可以设立三个“开关”（标志位），分别记录我们**有没有见过数字**、**有没有见过小数点**、**有没有见过 e/E**。
初始时它们都是 `False`。

1. **遇到数字 (0-9)**：
   * 把“见过数字”的开关设为 `True` (`has_digit = True`)。

2. **遇到加减号 (+, -)**：
   * 这个符号只能出现在**两个地方**：
     - 要么是整个队伍的第一位（如 `-1`）。
     - 要么紧紧跟在 `e` 或 `E` 的屁股后面（如 `1e-5`）。

3. **遇到小数点 (.)**：
   * 第一，一个数字里**不能有两个小数点**（即 `has_dot` 必须是 `False`）。
   * 第二，指数 `e` 后面必须是整数，所以**小数点不能出现在 e/E 的后面**（即 `has_e` 必须是 `False`）。
   * 如果通过，把开关打开：`has_dot = True`。

4. **遇到指数符号 (e, E)**：
   * 同理：指数符号**不能出现两次**（`has_e` 必须是 `False`）。
   * 而且，在 `e` 跑出来之前，前面必须得有数字（`has_digit` 必须是 `True`）。
   * 如果通过，把开关打开：`has_e = True`。
   * **⚠️ 最关键的一步**：遇到 `e` 之后，后面**必须**还得跟新的数字才算完整，所以我们要把“见过数字”的开关重置为 `False` (`has_digit = False`)。这样在最后检查时，如果没有新的数字出现（比如 `12e`），它就不合格。

5. **遇到其他字符**：
   * 直接判定非法。

**最后检查收尾**：
循环结束后，只要我们最后的状态是**见过数字** (`has_digit == True`)，它就是一个有效的数字！

**面试为什么选这个？**
因为它的推导过程非常顺滑。你在白板写代码时，可以和面试官有很好的互动口述：“我们现在来看小数点应该满足什么条件…”，这种逐条验证的方式容错率极高，不会出现大脑一片空白的情况。

## 3. 进阶理论法：确定性有限状态自动机 (DFA)

**核心思路：**
计算机底层的编译原理在做词汇分析时用的就是 DFA。DFA 的做法是构建一个「状态表」。我们定义出所有可能的状态（比如：起点、看到了符号、看到了整数、看到了 e……等），然后随着每个输入字符，去查表看应该跳转到什么新状态。

**优点：** 执行极度严谨，速度快，适合处理复杂的、可扩展的模式匹配。
**缺点：** 面试写这个简直是地狱难度！你需要背下来包含 9 种状态的转移矩阵表，稍不留神写错一个转移路径，整个代码就崩了，且很难当场肉眼 debug 出错在哪。因此，仅供作为“我很懂底层原理”的谈资。
